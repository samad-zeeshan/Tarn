02 — the measured optimization recorded run

Making the rollup faster, honestly

Two independent changes to the Spark rollup, isolated in a 2×2 and timed on an identical slice. The headline is the aggregation rewrite: Spark plans multiple COUNT(DISTINCT …) in one groupBy with an Expand, replaying every input row once per distinct expression before the shuffle. Collapsing to the distinct grain first removes it. Spark will not do that rewrite for you.

Why it is a real speedup and not a faster wrong answer
The check that makes the number claimable

All four variants were fingerprinted with an order-independent checksum (sum of per-row CRC32 over every output column). They agree exactly — so the speedup is a change in execution strategy, not in what was computed. A benchmark without this check is measuring the speed of being wrong.

The join result is reported with a caveat: its baseline is forced to a SortMergeJoin via autoBroadcastJoinThreshold=-1. Spark's AQE would broadcast the 749-row label table on its own, so that row measures the cost of a shuffle join — not a bug that was found and fixed. Saying so is the difference between a benchmark and a brag.

03 — realtime recorded from a local replay

Streaming windows, watermarked

Auth events replayed into Redpanda (Kafka API) and consumed by Spark Structured Streaming: 1-minute tumbling windows per identity, a watermark for late data, checkpointed, appended to Parquet that dbt reads as a streaming mart. The lag below is dominated by the watermark hold by design — a window cannot legally emit before it closes and the watermark passes it. A sub-second figure here would mean the watermark was not doing its job.

Sustained throughput and config
What the run actually did

This is an accelerated replay from a file into a single-broker Redpanda on the same laptop as the Spark driver. No network hop, no broker cluster, no competing load. It measures this pipeline's processing latency — not a production system's.

Choke points — the hosts that make the graph a small world
This is the finding that survived. Blast radius saturated because a handful of hubs are authenticated to by tens of thousands of distinct identities each — which collapses the network into a graph where almost everyone is two hops from almost everyone. Harden these and you break paths. That is an action item; "the attacker can reach almost every host in the estate" is not.

05 — a finding worth keeping

“Off-hours” had to be measured, not assumed

LANL ships relative seconds, not wall-clock time — nothing in the corpus says an event happened at 3am. The first attempt to derive an off-hours band found a nearly flat curve and refused to emit one. That was correct: machine accounts run around the clock and are most of the traffic. Split them out and the human cycle is unmistakable. The band the warehouse uses comes from this measurement — never from an assumed nine-to-five.

And then it caught nothing. This was the most carefully derived signal in the project, and as a detector it scored 0 of 181 compromised identity-days — see Q2 in the table above. Careful methodology does not entitle you to a result. Reporting the zero is the whole point; a portfolio that only shows the signals that worked is a brochure.
